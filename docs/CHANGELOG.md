108. Trigger/Automation refactor for autonomous watchers (spec `docs/specs/2026-06-19-trigger-abstraction.md`; migration 0011 drops `triggers`, creates `automations`).
     A **`Trigger`** is now a pluggable firing mechanism (`email`|`timer`|`webhook`) — a sibling abstraction to `Tool`/`LlmProvider` with a `paramsSchema` + `detect((params,cursor)) → {events,nextCursor}`, mapped name→impl by a worker `triggerRegistry`. An **Automation** is the configured row (chosen Trigger + params + objective + notify policy + cursor).
     Tier-0 detection is now each Trigger's own `detect()` returning an explicit list of new events, replacing the gate+reducer that collapsed everything to one opaque scalar (`gate`/`last_seen_signal` dropped; `params`/`cursor`/`pending_cursor` added; `kind`→`trigger`).
     The per-Trigger **cursor** is opaque to the framework (jsonb, stored + staged/committed, never interpreted). It commits **only after the escalated run reaches `done`** (`run.ts` does `cursor ← pending_cursor`); a crashed/failed run re-delivers the same delta next tick (at-least-once). This fixes email re-reporting/dropping — the old gate advanced the signal at *detection* time, silently losing mail when a run died.
     Agent-facing tools renamed: `create_automation` (was `schedule_self`) now picks a Trigger + supplies schema-validated params, killing the freeform-gate-or-null footgun; `run_automation` (was `run_trigger`) force-runs. Scratchpad scope `trigger:<id>`→`automation:<id>`.
     Operational names deliberately kept: pm2 `alfred-triggers`, pg-boss queue `trigger-detect`, `TriggerDetectJob`/`triggerId` (carries the automation id), `enqueueTriggerDetect`/`scheduleTrigger`/`unscheduleAll`/`bumpDetectionCost` — renaming them was churn for no design gain.
     Existing watcher rows are dropped (clean cutover, no FK references), re-created via `create_automation`. Build + lint green; tests 145 pass, 2 pre-existing local-DB-state failures (the documented `run.test.ts` approval-gate flake; `db.test.ts` needs `pnpm db:migrate` to create the new `automations` relation locally).

109. Trigger-abstraction adversarial-review fixes — the cursor commit is now regression-proof against interleaving (CHANGELOG 108 follow-up).
     **The dismiss path no longer regresses the cursor.** A non-deterministic triage can dismiss a delta a *prior* escalation already staged for an in-flight run; the dismiss `cursor`-advance cleared that run's `pending_cursor`, so the parked run's later done-commit wrote a now-stale staged value, regressing `cursor` and forcing re-triage of handled events. Dismiss + empty-delta now advance `cursor` only when no stage is in flight (`pending_cursor == null`), mirroring the empty-branch guard that already existed.
     **The done-commit is now conditional.** `run.ts` captures `pending_cursor` at run *start*; new `commitAutomationCursorIfStaged(id, expectedPendingCursor)` commits `cursor ← pending` only when the row's `pending_cursor` still equals the staged value (`= $val::jsonb`), so any concurrent writer that moved it wins and the stale commit is a no-op — never a regression, at worst a re-deliver (the safe direction). Replaces the unconditional `advanceAutomationCursor` at this one site (the helper stays for the dismiss-side direct commit).
     **Dismiss/empty cursor-advance + fire-record are now atomic** — folded into one `markAutomationFired({ cursor })` UPDATE (was two statements), so a crash between them can't re-detect the delta or double-charge `detection_cost_usd`.
     **Triage prompt: closed a fence-escape.** The untrusted events were wrapped in a markdown ```` ``` ```` code fence a crafted email subject/from could close early; switched to unbreakable prose markers (matching `composeObjective`). `STRICT_SYSTEM_PROMPT` remains the primary barrier; this is defense-in-depth.
     Rejected: exposing a `triage` arg on `create_automation` (review nit) — the spec's signature is exactly `{ trigger, params?, when?, objective, notify_policy? }`, so adding it would contradict the contract; triage stays settable only via direct rows. Added two `db.test.ts` cases reproducing the stale-commit regression + the atomic-fire advance (skip-guarded on `POSTGRES_URL`).
     Build + lint green; non-DB tests all pass. The 4 `pnpm test` failures are all pre-existing local-DB-state (3 × missing `automations` relation until `pnpm db:migrate` applies 0011, incl. the unchanged `trigger management` test; + the documented `run.test.ts` approval-gate flake) — none from these changes.

110. Email-Trigger param docs now explain the cursor-based dedup, so the agent stops mis-describing it.
     Live test surfaced the agent telling the owner that `unreadOnly:false` could re-report handled mail and that marking-read prevents retriggering — both false: dedup is the internal UID high-water-mark cursor (exactly-once per new message regardless of read state / `unreadOnly`); the read flag is irrelevant to retriggering. Root cause: the cursor is framework-internal/invisible to the agent, and the `email` `paramsSchema` copy (the only thing the agent sees) described `unreadOnly` as plain "only fire on unread", so the agent fell back on the unread-flag-as-dedup pattern this design replaced.
     Fix is copy-only: the `email` paramsSchema gained a top-level description (fires once per new message, dedup automatic, baseline silent) and `unreadOnly` is now documented as a *filter* — true ⇒ skips mail already read by poll time (a real miss-window), not a dedup knob. Also added a §16 read-only contract note on `Trigger.detect()` (the deleted gate's read-tier check no longer enforces it; read-only is now by-construction per impl). No behavior change.

111. Removed the `email` Trigger's `unreadOnly` param (supersedes the CHANGELOG-110 doc of it).
     It was a vestige of the pre-cursor model (where the unread flag *was* the dedup signal) and didn't survive the move to the UID cursor: dedup is already exactly-once via the high-water mark, so the only thing `unreadOnly:true` added was a racy miss-window (a new mail read before the next poll got filtered out AND the cursor advanced past it → silently skipped). It also filtered on volatile, human-mutated state, unlike the stable `from`/`subject`/`mailbox` filters, coupling detection to reading activity. "Watch my unread pile" is a different (state/threshold) trigger archetype, not a new-arrival feed filter.
     `email` params are now `{ mailbox?, from?, subject? }` (stable attributes only); `emailCriteria` no longer emits a `seen:false` IMAP criterion. No schema/migration change (`params` is freeform jsonb) and nothing in the DB referenced it.

120. New `update_automation` agent tool — edit an existing automation in place by id, so re-wording an objective or rescheduling a watcher no longer means delete-and-recreate (the gap flagged in CHANGELOG 119).
