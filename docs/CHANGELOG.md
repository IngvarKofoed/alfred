96. Included `alfred-discord` in the root `pnpm dev` concurrently set (it was deliberately left out at build time, CHANGELOG 95) — `dev` now runs worker/server/web/**discord** together. Safe in the `-k` (kill-others) set because the bot idles (never exits) when unconfigured; and hardened the boot path so a `client.login` failure — a bad/expired token, or (the likely first-run case) the privileged Message Content Intent not yet enabled in the developer portal — is caught, logged loudly, and **idles** rather than exiting, so a Discord login failure can't crash-loop the process under pm2 or take worker/server/web down with it under `concurrently -k`. `pnpm --filter @alfred/discord build` + `pnpm lint` green.

97. Discord: the bot's reply no longer pings the owner. The placeholder was sent via `message.reply('…')`, which Discord renders as a reply that **mentions** the author (the "@you" ping) — pure noise in a 1:1 DM. Now `message.reply({ content:'…', allowedMentions:{ repliedUser:false } })` (kept `message.reply` over `channel.send` because `message.channel`'s union includes `PartialGroupDMChannel`, which has no `.send`). `pnpm --filter @alfred/discord build` + `pnpm lint` green.

98. Wrote docs/specs/2026-06-17-discord-conversation-model.md — spec for the Discord conversation model (a brainstorm-with-codex+gemini → spec). Moves Discord from "one infinite DM = one ever-growing conversation" to a **private Alfred guild where a forum post is a conversation**: a forum post is a thread with its own channel id and the bot already keys conversations on `message.channelId`, so post-per-conversation drops onto the existing `(ingress, channel_key)` model with **zero schema change** (`channel_key = post/thread id`). The DM is **retired** — it just redirects to the guild, creating no conversation. Each recurring watcher becomes its own forum post too, which **collapses the notification problem**: a watcher posts into its thread, Discord's native push *is* the alert, and approvals are the same in-thread buttons — no separate notifier, no leaving Discord (Web Push stays for the web PWA, untouched). The load-bearing seam is the watcher→thread move: `run.ts` resolves the watcher's scratchpad by `conversation_id` (a new `getTriggerByConversation`) instead of `getTrigger(conv.channelKey)`, `detect.ts` routes each escalation through `triggers.conversation_id`, and a **bot-side watcher-reconcile** creates the posts (setting `conversation_id`) and proactively `LISTEN`s their channels up front (a watcher run has no preceding owner message). All gated on a new optional `DISCORD_GUILD_ID` — unset ⇒ today's DM-or-mention bot unchanged, so the change is additive. Settled the alternatives via the picker: sessionized DM (rejected — a "fake sidebar"), a doorbell `DiscordNotifier` (rejected — owner won't use the web app), and a read-only mirror seam (rejected — a reply wouldn't continue the watcher). Read-only spec phase — no code changed.

99. Built the Discord conversation model — guild forum threads (spec `docs/specs/2026-06-17-discord-conversation-model.md`; via a multi-agent workflow → adversarial 3-lens review → fix). Discord becomes a self-sufficient surface for a Discord-only owner: a private Alfred guild where a **forum post is a conversation** and each recurring watcher is its own post. **config:** new optional `DISCORD_GUILD_ID` — unset ⇒ today's DM-or-mention bot is byte-for-byte unchanged; set ⇒ the bot uses the forum-thread conversation model. **db:** `getTriggerByConversation(db, conversationId)` (the recurring-watcher lookup keyed on `triggers.conversation_id`, deterministic) + `setTriggerConversation(db, { id, conversationId })` (the bot's repoint). **worker (`detect.ts`):** `resolveConversationId` now routes a recurring watcher's escalation through `triggers.conversation_id` **when set** (the Discord post the bot created), else today's `trigger.id`/`ingress='trigger'` default — with a **NULL-guarded backfill** so the bot's repoint wins (the backfill only fills a still-NULL row, never overwrites the Discord conversation). Backward-compatible: an existing watcher with no `conversation_id` is unchanged and keeps its scratchpad (`memory_facts` scope `trigger:<id>`, keyed on the trigger id, so it survives the repoint). **worker (`run.ts`):** watcher resolution swapped from `getTrigger(conv.channelKey)` to `getTriggerByConversation(db, conv.id)`, keeping the `isTrigger` gate and a **recurring-only alias** so `'self'` one-shots and interactive runs are byte-for-byte unchanged (`human_in_loop=false` keeps `isTrigger` true regardless of ingress, so the §7.7 autonomous lifecycle is intact). **services/discord (the guild forum model, all gated on `DISCORD_GUILD_ID`):** find/create the `conversations` + `watchers` GuildForum channels; **mention-less** responses inside the Alfred guild (answer every owner message in a forum post, no `@` needed) + a one-line **DM redirect** (a DM creates no conversation/run); a bot-local **`/new [title]`** slash command (create a post + greeting); a **catch-all** conversation for stray in-guild messages outside a post; a **watcher-reconcile** (boot + 30s timer) that creates a `watchers` post per enabled watcher, repoints its `conversation_id`, and opens a standing `LISTEN` so the streamed run renders into the post; and **lazy watcher-render** — a watcher run has no user-message placeholder, so the bot posts a target message into the watcher's post on the run's first streamed event. Also a `standingListens` fix so an owner reply *in* a watcher post doesn't tear down the standing subscription. **Notifications become Discord-native** for the Discord-primary owner — a watcher posting into its thread is the alert; the Web Push outbox/dispatcher are off the critical path but untouched (still serve the web PWA). Post-review (adversarial 3-lens — blocker + should-fix + nit findings applied): the ungated `/self`-reminder notify-policy regression → keep `watcher` undefined for `'self'`; the detect backfill NULL-guard; harden `openWatcherTarget` to confirm a genuine watcher post in the Alfred guild; auto-archive hardening; a tag-required-forum log; and the deterministic `getTriggerByConversation`. **No schema change, no migration.** Verified: build/lint green; tests 157 passed / 1 pre-existing local-DB-state flake (the `run.test.ts` approval-gate, CHANGELOG 33/39/57/59/61/65/84) + 18 Discord helper tests. Live Discord behaviour (a real guild with Community enabled for forums + the Manage Channels permission) pending on-box verification.

100. Discord conversation-model refinements from live use (four owner-requested tweaks). (1) The bot now **writes plain messages instead of replying** — the placeholder is `message.channel.send('…')` (guarded by `isSendable()`, which narrows out the `.send`-less `PartialGroupDMChannel` that originally forced `message.reply`) rather than a reply-with-reference; in a 1:1 post the "replying to…" reference was pure noise (supersedes the CHANGELOG-97 ping-suppression). (2) **Tool usage is now visible in Discord** — each `tool_call_start` is woven into the streamed message as a quiet Discord-subtext chip (`-# 🔧 \`tool\` <short args>`, args truncated to ≤80 chars and only when the worker included them), mirroring the web client's inline chips. (3) **The forum post's title is the conversation title** — `getOrCreateConversationByChannel` gained an insert-only `title` (set on creation, never clobbering a later rename/auto-title); `onMessageCreate` passes the post's name for a thread, and the watcher-reconcile passes the watcher name, so a post-backed conversation is named by its post and the worker's auto-title is skipped. (4) **Discord conversations stay in the web history list but lose the "discord" badge** — by owner request the conversations remain visible (the all-ingress CHANGELOG-92 behavior is kept; `GET /api/conversations` is unchanged), but the web `Sidebar.tsx` no longer renders a `discord` tag on them (the `'discord'` case was dropped from `badgeLabel`; `trigger`→"watcher" and `voice`→"voice" tags stay). Touches `packages/db/src/queries.ts`, `services/discord/src/index.ts`, `clients/web/src/Sidebar.tsx`. `pnpm -r build` + `pnpm lint` green; no schema change.

102. iOS voice: faster time-to-first-word — dropped the on-device endpointing window `trailingSilenceToEnd` from **1.2s → 0.5s** (`clients/ios/.../VoiceController.swift`). That window is pure local dead time paid every turn: after the owner stops speaking, the app waits that long for silence before it endpoints and begins the STT upload, so it sits in front of the entire serial reply chain (upload → STT → run → LLM-to-first-sentence → first TTS clip). 0.5s is deliberately aggressive for snappiness; the comment notes to raise it if it endpoints on mid-sentence pauses (calibrate against the existing mic-level meter). No change to the cloud legs. Separately confirmed — **no code needed** — that **ElevenLabs flash TTS is already fully built and is a `.env` flip**: `ElevenLabsTtsProvider` already uses the low-latency `eleven_flash_v2_5` model with a default voice (`speech-provider.ts`), `makeTtsProvider()` selects it on `TTS_PROVIDER=elevenlabs`, the config schema has `ELEVENLABS_API_KEY`/`TTS_VOICE`, and iOS already decodes its MP3 (CHANGELOG 73/74); activating it is `TTS_PROVIDER=elevenlabs` + `ELEVENLABS_API_KEY` (+ optional ElevenLabs `TTS_VOICE` voice id) in `.env`, currently unset (Google defaults). `xcodebuild` (generic iOS Simulator) BUILD SUCCEEDED. 103. The `/speak` read-out command — speak the last assistant reply on demand, web (spec `docs/specs/2026-06-18-read-out-command.md`; built via a multi-agent /workflows run — additive foundation → 3 parallel disjoint consumers → verify — then a code-review high pass whose 3 should-fix + 6 nits were all applied). A web slash command (`/speak`, alias `/read`) takes the conversation's **last assistant message**, sentence-chunks it, synthesizes each chunk to a TTS clip in Alfred's configured voice, and plays it in the browser — **no agent run, no pg-boss job, no LLM loop**: a deliberately run-free, request/response path parallel to the run-coupled iOS voice pipeline (which is untouched). **Shared, ingress-agnostic core** so a future client (Discord/WhatsApp) reuses it by just delivering the clip: `writeBytesToWorkspace` + a new `writeAudioToWorkspace`/`AudioRef` and a `textFromContent` content-text joiner live in `@alfred/shared`; `stripMarkdownForSpeech`, `splitIntoSpeechChunks`, `synthesizeSpeech` (bytes-only, returns null on empty-after-strip) and `synthesizeToClip` (= synth + shared audio writer) in `@alfred/agent-core/src/speech.ts`; `readLastAssistantText` (newest assistant text, via the shared joiner) + `latestRunIdForConversation` in `@alfred/db`. **Web flow:** the `speak` command (pure `@alfred/db`-only registry) returns a new `CommandResult.action: 'speak'` directive; the client opens `POST /api/conversations/:id/speak`, which streams NDJSON `{seq,path,mimeType}` clip refs (Hono `streamText`) as each chunk synthesizes, and plays `/media` clips through an ordered `HTMLAudioElement` queue (the web client's first audio playback; epoch-guarded so a fresh `/speak` or a conversation switch stops the prior one). **Cost** is attributed to the conversation's most recent run via `recordOutOfLoopLlmCall` (an aggregated `'tts'` row, `/debug`-visible; an owner-chosen tradeoff — it doesn't re-sum into that finished run's rolled-up `cost_usd`). **Moves:** `writeAudioToWorkspace` relocated from `services/worker/src/audio.ts` (deleted) to `@alfred/shared`; the worker's `textOf` now delegates to `textFromContent`; both `stripMarkdownForSpeech` and `TTS_MIN_SENTENCE_CHARS` are imported from `@alfred/agent-core` (worker's live incremental flush behaviourally unchanged). **Code-review fixes:** the route now aborts synthesis on `c.req.raw.signal` (no billed TTS for a superseded/abandoned read-out); the `speak` command no longer pre-reads the message (the route's 422 is the single authority, so the message is loaded once — the web client shows a neutral "Nothing to read out yet." on 422 and "🔊 Speaking…" only once the stream is live); the error note is honest ("Some of that…" only if a clip already played); `splitIntoSpeechChunks` is a single forward pass (no per-iteration regex realloc / O(n²) re-slice); empty-after-strip chunks are skipped; and the web play queue removes clip listeners on advance + nulls the audio ref on drain. Web-only; Discord/iOS are deferred fast-follows on the same helpers. `pnpm -r build` (10 projects) + `pnpm lint` green; `pnpm test` 158 passed / 1 failed (the documented local-DB-state `run.test.ts` approval-gate flake, CHANGELOG 33/39/57/59/61/65/84), incl. a new `packages/db/src/db.test.ts` read-out-helpers round-trip. Live in-browser confirm of playback pending (per `clients/web/CLAUDE.md`).

105. `.env.example` → `.env` sync script (`scripts/sync-env.mjs`, run via `pnpm env:sync`). Cross-platform, pure-Node (no deps): rewrites `.env` to mirror `.env.example`'s line order, comments, and section headers while preserving the user's existing values — keys **added** to the example are added (carrying the example's placeholder + inline docs), keys **removed** from the example are removed only after per-key confirmation. Safety: secrets are never printed (only key NAMES appear in prompts/summaries), `.env` is copied to `.env.bak` before any write, an unchanged rebuild skips the write entirely, and a stale key is never deleted without an explicit `y` (an EOF/piped/non-interactive `AbortController`-aborted prompt defaults to keep). Stale keys the user keeps are appended under a `# === Not in .env.example (kept by sync-env) ===` header. Flags: `--dry-run` (plan only), `--yes` (delete all stale non-interactively), `--no-delete` (keep all stale), `--example`/`--env` path overrides, `--help`; first run with no `.env` just copies the example. `eslint.config.js` adds `AbortController` to the `scripts/**/*.mjs` globals. Verified with temp fixtures (add/delete/keep, order + value preservation, idempotency, first-run, `--yes`/`--no-delete`, CRLF preservation) and a `--dry-run` against the real repo files (15 example keys missing from the local `.env`, no stale); eslint clean. Code-review (high) applied: the output line-ending is taken from the target `.env` (not the example) so the no-op guard holds when the two files differ in LF/CRLF style (no needless rewrite), and the header documents the two by-design limitations — `.env`-only comments/blanks are replaced by the example's layout (values preserved; recover via `.env.bak`) and values are assumed single-line.
