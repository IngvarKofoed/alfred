# Alfred — Post-MVP Ingresses

Reference for the three non-web ingresses (Discord, voice, autonomous triggers). All three are now **built** (see their per-section notes below). Split out of `ARCHITECTURE.md` §9 (which keeps the ingress contract and the built web ingress, §9.1) so the always-loaded boot context stays small — read this when building or working on one of these surfaces.

All ingresses follow the same shape (§9): receive input → look up/create the conversation row → enqueue a job (pg-boss) → `LISTEN` on the conversation channel → forward `NOTIFY` payloads back → finalize on job complete. Section numbers reference `ARCHITECTURE.md`; the headings retain their original §9 numbers.

---

## 9.2 Discord bot (`alfred-discord`) — built

**Library**: **discord.js** (v14). The owner DMs (or @-mentions) the bot to reach the *same* Alfred — one identity, one memory, one tool set — from Discord. Spec: `docs/specs/2026-06-16-discord-bot.md` (build-order step 6, the end of MVP).

**Built (a direct-to-Postgres peer ingress, modelled on the webserver):** the bot creates runs + resolves interactions against the DB itself and `LISTEN`s the conversation channel for streamed output — exactly as the web ingress does — with **no dependency on `alfred-webserver` being up**. The inbound flow mirrors §9: `getOrCreateConversationByChannel(db, { ingress:'discord', channelKey: <discord channel id> })` → ingest image attachments → `createUserMessageRun` (one transaction) → register a per-conversation render state + `LISTEN "conversation:<id>"` → `enqueueAgentRun(runId)`. The run is created *before* the render state is registered, so a second message into a channel that already has an active run hits the one-active-run unique violation (23505) and short-circuits to "I'm still working on your last message" with no shared-state side effects (subscribe-before-enqueue still holds — the worker emits tokens only after `boss.send`).

- **Owner-only** (§12): every gateway message + every interaction is dropped unless `author.id === ALLOWED_DISCORD_USER_ID` — being the owner *is* the auth. Intents `Guilds, GuildMessages, DirectMessages, MessageContent` (+ `Partials.Channel` for DMs); MessageContent is a **privileged** intent (enable it in the Discord developer portal).
- **One persistent conversation per Discord channel** — originally via `unique(ingress, channel_key)` with `channel_key = <discord channel id>`; now (multi-surface spec 2026-06-21, after that index was dropped) via a `conversation_surfaces(surface='discord', external_key=<channel id>)` binding row, so a DM/guild channel/post is a continuous thread across days — surfaced, titled (ingress-agnostic `maybeAutoTitle`), in the unified web/iOS history (the all-ingress `GET /api/conversations`).
- **Streaming:** one shared `pg.Client` LISTENs every active run's channel and dispatches NOTIFY payloads by channel name to the matching render state (serialized per-conversation on an `eventChain`). A trailing-edge **~1.5s edit throttle** coalesces token bursts into one message edit (well under Discord's ~5-edits/5s/channel limit), always flushing the final state on `done`/`error`/`cancelled`. A reply growing past Discord's **2000-char** message limit seals the trailing message at the hard limit and opens a continuation message as the new edit target.
- **Approvals:** a `write`/`destructive` tool call (`interaction_required`, kind `approval`) posts a message with the proposed action + **Approve/Reject buttons** (the `interactionId` rides the `customId`).
- **Questions (`ask_user`):** kind `question` renders a **StringSelectMenu** for the options (single/multi per `multi_select`) and/or an "Other…"/"Answer…" button opening a **freeform Modal** (per `allow_freeform`) — native Discord components, mirroring the web client's "Other" choice.
- **Shared resolve writer:** a unified `interactionCreate` dispatcher routes buttons/selects/modals (and slash commands) by the embedded `interactionId`, resolving through `resolveInteraction(db, id, { response, resolvedVia:'discord' })` — the first-writer-wins conditional `UPDATE … WHERE status='pending'` + `interaction_resolved` NOTIFY, **shared with the webserver** (its `writeResolution` is now a thin caller). Kind-agnostic, so the same writer serves `{ approved }` and `{ selected_labels, freeform_text }`. A lost race → "Already resolved"; an `interaction_resolved` from another surface (answered on web, or timed out) disables the components here too.
- **Images both ways via the conversation workspace + content parts:** inbound attachments (≤10 MB, image mime only) download into the workspace as `{ type:'image', path, mimeType }` parts (extension derived from the *validated* mime, never the client filename); outbound, on `done`, the run's newer `assistant`/`tool` image parts are read off disk and uploaded as Discord attachments. No image bytes ride NOTIFY; no agent-core change.
- **Commands as native Discord slash commands** over the new **`@alfred/commands`** package: the backend command registry (`rename`, `help`) moved out of the webserver into a shared package, so the bot registers Discord *global* slash commands from `listCommands()` and dispatches them to the same `executeCommand` the web `POST /commands` route uses — one registry, no drift; the result is an ephemeral reply (the `conversation.title` echo is web-only).

Two new `@alfred/db` helpers underpin this: `getOrCreateConversationByChannel` (map a Discord channel to its long-lived conversation — originally by the natural `(ingress, channel_key)` key; now, multi-surface spec 2026-06-21, via the `conversation_surfaces` binding table, see below) and `resolveInteraction` (the shared interaction-resolve writer above). New config keys `DISCORD_BOT_TOKEN` + `ALLOWED_DISCORD_USER_ID` (optional); unconfigured, the process logs and idles rather than crash-looping under pm2 (mirrors the updater's inert-unless-enabled shape). **Fail-and-restart** (§7.6): the bot holds no durable mid-run state — a restart mid-run loses the in-flight render (the run still completes server-side, visible in the web app) — but it DOES re-establish a standing `LISTEN` per **bound** conversation at boot (from `conversation_surfaces`, multi-surface spec 2026-06-21), so a bound conversation's future runs keep streaming into its Discord post across a restart.

**Built (the guild forum conversation model — spec `docs/specs/2026-06-17-discord-conversation-model.md`):** gated on a new optional `DISCORD_GUILD_ID`, the bot becomes a self-sufficient surface for a Discord-only owner — moving from "one infinite DM = one ever-growing conversation" to a **private Alfred guild where a forum post is a conversation**. Unset ⇒ the DM-or-@-mention bot above is byte-for-byte unchanged; the whole model is additive.

- **Forum post = conversation.** A forum post is a thread with its own channel id, and `getOrCreateConversationByChannel` keys on `message.channelId` — so each post resolves to its own conversation (history, runs, auto-title, cost rollup). Resolution originally rode the `(ingress, channel_key)` model with `channel_key = <post/thread id>`; with the multi-surface model (spec 2026-06-21) that index is gone and the post→conversation mapping is a `conversation_surfaces` row (so a *web/iOS-born* conversation can also gain a post — see "pull to Discord" below). The bot finds/creates two **GuildForum** channels: `Conversations` (each post a conversation; start one with Discord's native New Post, the bot-local **`/new [title]`** slash command — which posts a greeting — or **`/pull`**, which materializes an existing web/iOS conversation as a post) and `Watchers` (one post **per fire**, created on demand from the notifications outbox — see below). The forum's post list + native archive/tags is the conversation sidebar.
- **Mention-less inside the Alfred guild; the DM is retired.** In a guild whose id matches `DISCORD_GUILD_ID` the bot answers **every** owner message in a forum post with no `@`-mention required; a **DM** gets a one-line "let's talk in your Alfred server" pointer and creates no conversation/run; any *other* guild keeps the existing DM-or-mention filter. A **catch-all** conversation is the safety net for a stray in-guild message landing outside a post (so it still gets a reply, not a drop).
- **A post per fire, created from the notifications outbox (per-fire spec `docs/specs/2026-06-20-watcher-conversation-per-fire.md`; supersedes the persistent-watcher-post model above).** A watcher no longer owns one ever-growing post. Each Tier-2 escalation lands in its **own fresh conversation** (the worker's `createAutomationRun` mints it `ingress='trigger'`, `conversations.automation_id` set, an Alfred-first hidden seed; ARCHITECTURE §9.4) and writes a `notifications` row on `done`. The bot becomes a **second consumer of the `notifications` channel** (alongside the worker's Web Push dispatcher — `LISTEN/NOTIFY` broadcasts to both): on the **first** notification for a watcher conversation (a mid-run `approval`/`question`, or the final `result`/`error`) it **creates a new `Watchers`-forum post** — a new thread *is* a real Discord ping — posts the report read from the conversation's final assistant message (text + any workspace images, post-final, **no token streaming** — the autonomous run renders post-final from the DB, *not* via a live LISTEN, since the post itself is the alert), and **binds** the conversation to the post via `bindConversationSurface` (a `conversation_surfaces(discord, post id)` row — multi-surface spec 2026-06-21, replacing the old `repointConversationChannel`: the conversation keeps `ingress='trigger'` as origin and merely gains a Discord presence). A separate `watcherRenderByConversation` map holds the post-final approval/question render state, kept apart from the live-streaming `renderByChannel` map so the two never clobber each other across an approval→result fire. The worker stays Discord-agnostic; the bot reacts purely to the outbox.
- **Reply resolution + the standing LISTEN.** Because the post is now bound (not repointed), a **reply in the post continues the conversation** through the normal `onMessageCreate` path — which resolves the post via `getConversationBySurface` (the binding), not `channel_key`. The owner's message is the trigger for a *Discord-originated* turn. For a **web/iOS-originated** run on a bound watcher conversation, the bot's **standing LISTEN** (multi-surface spec 2026-06-21, re-established at boot from `conversation_surfaces`) live-streams it into the post — re-adding for the *curated, explicitly-bound* set what the per-fire change removed wholesale (there's no per-watcher reconcile). A web reply to a *just-fired* watcher before the next boot still lands in the DB + on web but won't live-mirror to Discord that session — an accepted gap (watcher conversations lean post-final). Notification dedup is in-process (`seenNotifs`, seeded at boot with the existing backlog so historical fires get **no** catch-up post — a fire while the bot is down notifies via Web Push only).
- **Notifications are Discord-native and cross-surface.** Both surfaces fire for a watcher result: the Discord post (always, when the guild is configured) and Web Push (if subscribed) deep-link to the same conversation, and either surface's reply continues it. The owner sees the report wherever they look; Approve/Reject buttons + question components render in the post, so in-thread actions resolve the interaction without leaving Discord.

**Built (multi-surface conversations — pull any conversation into Discord; spec `docs/specs/2026-06-21-multi-surface-conversations.md`):** one logical conversation the owner can start on web/iOS and continue in Discord (and back) — Discord presence is now an explicit, lazily-created *materialization*, not a property of where the conversation was born. The load-bearing change is the routing move from `conversations.channel_key` (its `unique(ingress, channel_key)` index dropped, migration 0013) to the `conversation_surfaces` binding table (DATABASE.md): a Discord post → its canonical conversation is a `(surface='discord', external_key=postId)` row. `ingress` stays as origin metadata (still drives `human_in_loop`); the conversation id never changes, so web deep-links + the sidebar are untouched, and web/iOS keep showing full all-ingress history with **no change** (they already aggregate all ingresses — there was nothing to mirror *to* web). v1 is web-family + Discord only (iOS rides the web API); a general N-surface registry / `SurfaceProjector` is deferred.

- **"Continue in Discord" is explicit + lazy** — never automatic, so ephemeral web chats never clutter the guild. Two triggers: a web **button** (a thin "💬 Continue in Discord" affordance above the chat composer → `POST /api/conversations/:id/surfaces/discord`) and a Discord **`/pull`** slash command (a `StringSelectMenu` picker of recent *not-yet-bound* conversations, `listUnboundConversations`). The webserver has no Discord client, so the button is **fire-and-forget over a `discord-pull` NOTIFY channel** (`requestDiscordPull`) — the bot LISTENs it and materializes out of band (so the web side learns no post id, just an accept ack; idempotent — a 409 if already bound). `/pull` materializes **directly** in-process so the ephemeral reply can carry the post link. Materialization (`materializePull`, serialized on a `pullChain`): create a `Conversations`-forum post (reusing the watcher path's `createForumPost`), write the `conversation_surfaces` binding, **mirror the last ~20 messages** into the post (chronological, with an "_… earlier on web →_" note when older history was truncated, so it reads as a real thread without a full backfill / rate-limit risk), and open a standing LISTEN. From then on it's an ordinary bound conversation.

- **A bound conversation streams live + mirrors web/iOS turns in.** The bot re-subscribes to every bound conversation's channel at boot (`listBoundConversations` → `ensureStandingListen`, a persistent `renderByChannel` state with `standing=true` that survives `done` rather than being torn down) and renders its runs live into the post via the existing interactive `renderByChannel` path. For a web/iOS-originated run there's no `onMessageCreate` placeholder, so on the run's **first** streamed event the bot lazily prepares the post (`prepareStandingRun`): mirror the owner's web/iOS **user** turn in as a quoted "🧑 …" line, then open the assistant reply's render target. A Discord-*originated* reply is handled by the normal `onMessageCreate` path — which, on a bound channel, reuses the standing state instead of opening a new per-turn LISTEN. **`message_surface_refs`** (DATABASE.md) is the dedup ledger throughout: the bot records a ref for every message it mirrors (pull history, each user turn, each run's assistant/tool turns) and checks `hasMessageSurfaceRef` before mirroring, so it never re-posts a message already present (its own renders, or a Discord-originated turn) — no separate bridge process.

- **Watcher per-fire conversations fold into this model.** A watcher conversation is now just "a conversation auto-bound to Discord" — the per-fire path's `bindConversationSurface` (above) is the autonomous variant of pull-to-Discord's explicit bind, and `automations.conversation_id` is unchanged. Its autonomous run still renders **post-final** (no standing LISTEN opened during that run — opening one would double-render it live); only its bound *replies* (and web/iOS-originated follow-ups, from the next boot's standing LISTEN) stream.

- **Invariants unchanged.** One active run per `conversation_id` still holds (the partial unique index), so a second surface's input while busy hits the existing busy-reject ("Alfred is working"). Approvals/questions already broadcast via `user_interactions` + the `interaction_required` NOTIFY, so a bound post shows the buttons and the web shows the card simultaneously, and the first-responder conditional UPDATE resolves it while the other surface tears down — it just works once bound, no new machinery.

**Non-goals (v1, both models):** in-Discord cancellation (free a stuck conversation from the web Stop button), the approval "don't ask again" flag, non-image attachments, in-thread context summarization (a single post can still overflow and fails loudly, as web — the answer is "start a new post"), migrating the old DM history into the guild, and auto topic-segmentation (the owner starts posts).

## 9.3 Voice (`alfred-voice`) — native app only, post-MVP

Hands-free is required and **only the native app** is a voice surface (the PWA is chat-only). All components are **cloud APIs** (likely ElevenLabs/Google TTS, Deepgram/Google STT — final choice deferred), so the orchestrator just routes audio+text streams: `app mic → WS → STT → agent worker (pg-boss + NOTIFY) → TTS → WS → app speaker`. Key points: API keys live **server-side, never in the client** (or anyone could spend the owner's budget); the agent core stays the brain (no lock-in to a realtime voice API); wake word runs **on-device** (Picovoice Porcupine or openWakeWord) and only then opens the upstream WS. Native platform deferred — its only contract is "WebSocket with bidirectional audio + a control channel."

**Built (Approach B — voice as an I/O modality on the text pipeline, not a WS orchestrator):** the iOS text-chat client now has hands-free voice without any of the above — no `services/voice` process, no WebSocket. STT is a request/response REST route (`POST /api/conversations/:id/audio`, ARCHITECTURE §9.1) that posts the transcript as a normal user message + a `speak` run; TTS is synthesized server-side by the worker as the run streams (sentence-chunked) and pushed back over the **existing SSE channel** as `tts_audio` events referencing `/media` audio clips (ARCHITECTURE §6.2). Provider-swappable (Google/ElevenLabs) via the agent-core `SttProvider`/`TtsProvider` abstractions (§7.2), keys server-side. On-device capture/AEC/VAD/barge-in live in the app's `VoiceController`. Spec: `docs/specs/2026-06-14-voice-stt-tts.md`. The WebSocket orchestrator above remains the **deferred Approach C** upgrade path if sentence-granular latency ever proves inadequate.

## 9.4 Autonomous triggers (`alfred-triggers`) — built

Triggers are ingresses with **no human at the other end** — they enqueue jobs on time/events; results are logged or pushed as a notification through an interactive ingress. Categories: **scheduled** (cron-style, e.g. "8am inbox summary"), **event-driven** (inbox watchers, webhooks, FS watchers — normalize the event, enqueue), **agent-initiated** (the agent creates a future automation; a Postgres row picked up by the scheduler). All share the same `enqueue + LISTEN + optionally notify` *transport* as human ingresses; the *execution lifecycle* differs (§7.7 / RUNTIME.md) so adding triggers was wiring, not a redesign.

**Trigger / Automation model (trigger-abstraction spec `docs/specs/2026-06-19-trigger-abstraction.md`, refines the autonomous-watchers build below):** two named concepts. A **`Trigger`** is the pluggable *firing mechanism* — `email`, `timer`, `webhook` — a sibling abstraction to `Tool`/`LlmProvider` in `agent-core` (pure types; concrete Triggers live in the worker, where Tools + providers already are). Each declares `name`, `mode` (`poll`/`push`), a `paramsSchema` (JSON Schema, validated at create time like `Tool.inputSchema`), and `detect((params, cursor)) → { events, nextCursor }` — owning its own deterministic delta-detection. An **Automation** is a configured instance — a chosen Trigger + its params + the action (`objective`) + notify policy + cursor state — which is the row in the renamed **`automations`** table (was `triggers`; `kind`→`trigger`, `gate`/`last_seen_signal` dropped, `params`/`cursor`/`pending_cursor` added; DATABASE.md). A worker-side `triggerRegistry` maps `trigger` name → `Trigger`, looked up in `detect.ts` the way the tool catalog is looked up.

**The cursor replaces the scalar `last_seen_signal`.** The old Tier-0 was a `gate` + reducer that diffed a read-tier `Tool` result down to one opaque scalar (`maxUid`/`count`/`hash`) — it answered *did anything change?* but never *which discrete items are new, handled exactly once?*. Now Tier 0 is the Trigger's `detect()`, which returns an explicit **list of new events** (each normalized to `{ id, summary, data? }`) plus an advanced **cursor**. The framework never *interprets* the cursor — it stores it as jsonb and stages/commits it; all delta semantics live inside `detect()` (so a new Trigger scales by choosing its own cursor shape: scalar high-water mark, opaque sync token, windowed seen-set, last-state edge, or none). The `email` Trigger is a monotonic-feed example: a `{ lastUid }` cursor over a UID-floor IMAP search (`fetchSinceUid`, bypassing the agent Tool layer since `list_emails` has no UID floor); first detect establishes the baseline silently (no fire on the pre-existing backlog).

**Cursor commits *after* the action run, not at detection — the core fix.** On escalation, `detect.ts` **stages** `pending_cursor = nextCursor` and leaves `cursor` untouched; `run.ts` commits `cursor ← pending_cursor` only when the escalated run reaches terminal `done` **and** the run is the autonomous escalation (`human_in_loop=false`) — a follow-up *reply* in a per-fire conversation (`human_in_loop=true`) resolves the same automation for its scratchpad but never staged a cursor, so it is side-effect-free on detection state. A crashed/failed/cancelled run leaves both untouched, so the next detect tick re-reads the same `cursor`, recomputes the same delta, and re-escalates — **at-least-once**. (On a Tier-1 dismiss there's no run to wait for, so `detect.ts` commits straight to `cursor`.) This fixes email re-reporting: the old `gate` advanced the signal at *detection* time, silently dropping mail if the run died before reporting it. The fenced new-events delta (`{ uid, from, subject }`, untrusted watched content per §16) reaches triage and the Tier-2 objective, so the run knows *which* items are new and reads full bodies by uid.

**Built (autonomous watchers — spec `docs/specs/2026-06-16-autonomous-watchers.md`, refined by the model above):** A pm2 process **`alfred-triggers`** is a *pure scheduler* — it loads enabled `automations` rows at boot, registers cron schedules on the existing pg-boss queue (and sweeps one-shot automations — a `nextFireAt` with no recurring `schedule`, e.g. the old `self` reminder, now a `timer` automation — for their due time), and enqueues a **`trigger-detect`** job per due tick (queue/job-payload names kept operational; the `triggerId` field carries the `automations.id`). It decides *when* purely from each row's `trigger` + `schedule`/`next_fire_at` and never imports the `triggerRegistry` or tool layer (all detection + tool execution stays in the worker). The worker's detect handler runs the **tiered detection ladder**: Tier 0 the Trigger's `detect()` (empty `events` ⇒ free idle path, no fire), Tier 1 a cheap-model triage over the events (`DETECTION_MODEL`, `escalate`/`dismiss`), Tier 2 — only on escalation — a full action run created via the unchanged `agent_runs` + `boss.send('agent-run')` path with `human_in_loop=false`. **Detection writes no `agent_runs` row**, so idle polls cost nothing and don't litter the run log. **Each escalation gets its own fresh conversation (per-fire spec `docs/specs/2026-06-20-watcher-conversation-per-fire.md`):** the worker's `createAutomationRun` mints a new `conversations` row (`ingress='trigger'`, `conversations.automation_id` set) with an **Alfred-first hidden seed** — the objective + fenced delta ride `trigger_context` content parts the renderers skip, so the model still sees them but Alfred's report is the first *visible* message, repliable to continue. (This replaced the old persistent-thread model where every fire appended to one growing conversation.) The routing key is `conversations.automation_id` (run.ts: `getAutomationForConversation`), not the old `automations.conversation_id` (now repurposed to a "jump to latest" pointer), since a watcher now has many conversations. Continuity is the **objective scratchpad** (`memory_facts` scope `automation:<id>`, folded into the run's `[system]` block, §6.4/§7.7) — keyed on the automation id, so it survives across the per-fire conversations and a follow-up reply recalls it. The agent's **`create_automation`** tool (was `schedule_self`) self-schedules future runs — picking a Trigger and supplying schema-validated params, killing the freeform-`gate`-or-null footgun that fired email watchers every tick — while **`run_automation`** (was `run_trigger`) force-runs an existing one for testing (§7.7 seam 3), and **`update_automation`** edits an existing one in place (any subset of name/objective/params/`when`/notify_policy/enabled, keyed by id; write-tier like `disable_trigger`/`delete_trigger` so a prompt-injected agent can't silently repurpose a watcher — changing the trigger *mechanism* still means delete-and-recreate). Because no client is watching when a watcher fires, results/approvals reach the owner via a **durable notifications outbox** (the `notifications` table + the separate `notifications` NOTIFY channel; deep-linking into the per-fire `/conversation/:id` chat + approval UI), not SSE — which is dropped on a disconnected client. Two consumers now LISTEN that channel independently: the worker dispatcher → **Web Push** (the agent-core `Notifier` / `WebPushNotifier`, §7.2/§6.2) and the **Discord bot**, which posts a `Watchers`-forum post per fire (§9.2). Deferred (Non-goals): a generic windowed seen-set for non-monotonic feeds, an `rss` Trigger, exactly-once delivery, native iOS APNs, other transports (ntfy/Discord DM), the full §7.7 layered objective/daily budget, and an automation-management UI (v1 manages them via the `create_automation`/`update_automation`/`disable_trigger`/`delete_trigger` tools + direct rows).
