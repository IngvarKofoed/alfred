# Alfred — Post-MVP Ingresses

Reference for the three non-web ingresses (Discord, voice, autonomous triggers). All three are now **built** (see their per-section notes below). Split out of `ARCHITECTURE.md` §9 (which keeps the ingress contract and the built web ingress, §9.1) so the always-loaded boot context stays small — read this when building or working on one of these surfaces.

All ingresses follow the same shape (§9): receive input → look up/create the conversation row → enqueue a job (pg-boss) → `LISTEN` on the conversation channel → forward `NOTIFY` payloads back → finalize on job complete. Section numbers reference `ARCHITECTURE.md`; the headings retain their original §9 numbers.

---

## 9.2 Discord bot (`alfred-discord`) — built

**Library**: **discord.js** (v14). The owner DMs (or @-mentions) the bot to reach the *same* Alfred — one identity, one memory, one tool set — from Discord. Spec: `docs/specs/2026-06-16-discord-bot.md` (build-order step 6, the end of MVP).

**Built (a direct-to-Postgres peer ingress, modelled on the webserver):** the bot creates runs + resolves interactions against the DB itself and `LISTEN`s the conversation channel for streamed output — exactly as the web ingress does — with **no dependency on `alfred-webserver` being up**. The inbound flow mirrors §9: `getOrCreateConversationByChannel(db, { ingress:'discord', channelKey: <discord channel id> })` → ingest image attachments → `createUserMessageRun` (one transaction) → register a per-conversation render state + `LISTEN "conversation:<id>"` → `enqueueAgentRun(runId)`. The run is created *before* the render state is registered, so a second message into a channel that already has an active run hits the one-active-run unique violation (23505) and short-circuits to "I'm still working on your last message" with no shared-state side effects (subscribe-before-enqueue still holds — the worker emits tokens only after `boss.send`).

- **Owner-only** (§12): every gateway message + every interaction is dropped unless `author.id === ALLOWED_DISCORD_USER_ID` — being the owner *is* the auth. Intents `Guilds, GuildMessages, DirectMessages, MessageContent` (+ `Partials.Channel` for DMs); MessageContent is a **privileged** intent (enable it in the Discord developer portal).
- **One persistent conversation per Discord channel** via `unique(ingress, channel_key)` with `channel_key = <discord channel id>`, so a DM or guild channel is a continuous thread across days — surfaced, titled (ingress-agnostic `maybeAutoTitle`), in the unified web/iOS history (the all-ingress `GET /api/conversations`).
- **Streaming:** one shared `pg.Client` LISTENs every active run's channel and dispatches NOTIFY payloads by channel name to the matching render state (serialized per-conversation on an `eventChain`). A trailing-edge **~1.5s edit throttle** coalesces token bursts into one message edit (well under Discord's ~5-edits/5s/channel limit), always flushing the final state on `done`/`error`/`cancelled`. A reply growing past Discord's **2000-char** message limit seals the trailing message at the hard limit and opens a continuation message as the new edit target.
- **Approvals:** a `write`/`destructive` tool call (`interaction_required`, kind `approval`) posts a message with the proposed action + **Approve/Reject buttons** (the `interactionId` rides the `customId`).
- **Questions (`ask_user`):** kind `question` renders a **StringSelectMenu** for the options (single/multi per `multi_select`) and/or an "Other…"/"Answer…" button opening a **freeform Modal** (per `allow_freeform`) — native Discord components, mirroring the web client's "Other" choice.
- **Shared resolve writer:** a unified `interactionCreate` dispatcher routes buttons/selects/modals (and slash commands) by the embedded `interactionId`, resolving through `resolveInteraction(db, id, { response, resolvedVia:'discord' })` — the first-writer-wins conditional `UPDATE … WHERE status='pending'` + `interaction_resolved` NOTIFY, **shared with the webserver** (its `writeResolution` is now a thin caller). Kind-agnostic, so the same writer serves `{ approved }` and `{ selected_labels, freeform_text }`. A lost race → "Already resolved"; an `interaction_resolved` from another surface (answered on web, or timed out) disables the components here too.
- **Images both ways via the conversation workspace + content parts:** inbound attachments (≤10 MB, image mime only) download into the workspace as `{ type:'image', path, mimeType }` parts (extension derived from the *validated* mime, never the client filename); outbound, on `done`, the run's newer `assistant`/`tool` image parts are read off disk and uploaded as Discord attachments. No image bytes ride NOTIFY; no agent-core change.
- **Commands as native Discord slash commands** over the new **`@alfred/commands`** package: the backend command registry (`rename`, `help`) moved out of the webserver into a shared package, so the bot registers Discord *global* slash commands from `listCommands()` and dispatches them to the same `executeCommand` the web `POST /commands` route uses — one registry, no drift; the result is an ephemeral reply (the `conversation.title` echo is web-only).

Two new `@alfred/db` helpers underpin this: `getOrCreateConversationByChannel` (map a channel to its long-lived conversation by the natural `(ingress, channel_key)` key — `ensureConversation` is keyed by the PK and can't) and `resolveInteraction` (the shared interaction-resolve writer above). New config keys `DISCORD_BOT_TOKEN` + `ALLOWED_DISCORD_USER_ID` (optional); unconfigured, the process logs and idles rather than crash-looping under pm2 (mirrors the updater's inert-unless-enabled shape). **Fail-and-restart** (§7.6): the bot holds no durable state — a restart mid-run loses the in-flight render (the run still completes server-side, visible in the web app); the guild-model reconcile below re-establishes its standing watcher listens on boot.

**Built (the guild forum conversation model — spec `docs/specs/2026-06-17-discord-conversation-model.md`):** gated on a new optional `DISCORD_GUILD_ID`, the bot becomes a self-sufficient surface for a Discord-only owner — moving from "one infinite DM = one ever-growing conversation" to a **private Alfred guild where a forum post is a conversation**. Unset ⇒ the DM-or-@-mention bot above is byte-for-byte unchanged; the whole model is additive.

- **Forum post = conversation, zero schema change.** A forum post is a thread with its own channel id, and `getOrCreateConversationByChannel` already keys on `message.channelId` — so each post resolves to its own conversation (history, runs, auto-title, cost rollup) onto the existing `(ingress, channel_key)` model with `channel_key = <post/thread id>`. The bot finds/creates two **GuildForum** channels: `Conversations` (each post a conversation; start one with Discord's native New Post or the bot-local **`/new [title]`** slash command, which posts a greeting) and `Watchers` (one post per watcher). The forum's post list + native archive/tags is the conversation sidebar.
- **Mention-less inside the Alfred guild; the DM is retired.** In a guild whose id matches `DISCORD_GUILD_ID` the bot answers **every** owner message in a forum post with no `@`-mention required; a **DM** gets a one-line "let's talk in your Alfred server" pointer and creates no conversation/run; any *other* guild keeps the existing DM-or-mention filter. A **catch-all** conversation is the safety net for a stray in-guild message landing outside a post (so it still gets a reply, not a drop).
- **A watcher's conversation IS its Discord post (the load-bearing seam).** A recurring watcher's persistent conversation moves from `ingress='trigger', channel_key=trigger.id` to a Discord post (`ingress='discord', channel_key=post.id`), with `triggers.conversation_id` pointing at it. The worker stays Discord-agnostic; the seam is three changes: `run.ts` resolves the watcher by `getTriggerByConversation(db, conv.id)` (new query keyed on `triggers.conversation_id`) instead of `getTrigger(conv.channelKey)` — keeping the `isTrigger` gate and a recurring-only alias so `'self'` one-shots + interactive runs are unchanged; `detect.ts` `resolveConversationId` routes each escalation through `triggers.conversation_id` **when set** (else today's `trigger.id`/`ingress='trigger'` default), with a NULL-guarded backfill so the bot's repoint wins; and the scratchpad scope stays `memory_facts` scope `trigger:<id>` (keyed on the trigger id), so continuity survives the repoint. An existing watcher with no `conversation_id` is unchanged.
- **The bot proactively listens (watcher-reconcile).** A watcher run has no preceding owner message, so the bot subscribes up front: a reconcile (at boot + on a 30s timer, mirroring how `alfred-triggers` reconciles cron) creates a `Watchers` post per enabled watcher when missing, repoints `triggers.conversation_id`, and opens a standing `LISTEN` on its conversation channel. Because there's no user-message placeholder, watcher output is **lazily rendered** — the bot posts a target message into the watcher's post on the run's *first* streamed event, then streams/edits into it via the same paths as a normal turn. (A `standingListens` fix keeps an owner reply *in* a watcher post from tearing down that standing subscription.)
- **Notifications become Discord-native.** With watcher output + Approve/Reject buttons landing in the watcher's post, Discord's own push *is* the alert and the in-thread buttons action it — no separate notifier, no leaving Discord. The `notifications` outbox + dispatcher + Web Push are off the critical path for the Discord-primary owner but **untouched** (they still serve the web PWA).

**Non-goals (v1, both models):** in-Discord cancellation (free a stuck conversation from the web Stop button), the approval "don't ask again" flag, non-image attachments, in-thread context summarization (a single post can still overflow and fails loudly, as web — the answer is "start a new post"), migrating the old DM history into the guild, and auto topic-segmentation (the owner starts posts).

## 9.3 Voice (`alfred-voice`) — native app only, post-MVP

Hands-free is required and **only the native app** is a voice surface (the PWA is chat-only). All components are **cloud APIs** (likely ElevenLabs/Google TTS, Deepgram/Google STT — final choice deferred), so the orchestrator just routes audio+text streams: `app mic → WS → STT → agent worker (pg-boss + NOTIFY) → TTS → WS → app speaker`. Key points: API keys live **server-side, never in the client** (or anyone could spend the owner's budget); the agent core stays the brain (no lock-in to a realtime voice API); wake word runs **on-device** (Picovoice Porcupine or openWakeWord) and only then opens the upstream WS. Native platform deferred — its only contract is "WebSocket with bidirectional audio + a control channel."

**Built (Approach B — voice as an I/O modality on the text pipeline, not a WS orchestrator):** the iOS text-chat client now has hands-free voice without any of the above — no `services/voice` process, no WebSocket. STT is a request/response REST route (`POST /api/conversations/:id/audio`, ARCHITECTURE §9.1) that posts the transcript as a normal user message + a `speak` run; TTS is synthesized server-side by the worker as the run streams (sentence-chunked) and pushed back over the **existing SSE channel** as `tts_audio` events referencing `/media` audio clips (ARCHITECTURE §6.2). Provider-swappable (Google/ElevenLabs) via the agent-core `SttProvider`/`TtsProvider` abstractions (§7.2), keys server-side. On-device capture/AEC/VAD/barge-in live in the app's `VoiceController`. Spec: `docs/specs/2026-06-14-voice-stt-tts.md`. The WebSocket orchestrator above remains the **deferred Approach C** upgrade path if sentence-granular latency ever proves inadequate.

## 9.4 Autonomous triggers (`alfred-triggers`) — built

Triggers are ingresses with **no human at the other end** — they enqueue jobs on time/events; results are logged or pushed as a notification through an interactive ingress. Categories: **scheduled** (cron-style, e.g. "8am inbox summary"), **event-driven** (inbox watchers, webhooks, FS watchers — normalize the event, enqueue), **agent-initiated** (the agent schedules a future trigger; a Postgres row picked up by the scheduler). All share the same `enqueue + LISTEN + optionally notify` *transport* as human ingresses; the *execution lifecycle* differs (§7.7 / RUNTIME.md) so adding triggers was wiring, not a redesign.

**Built (autonomous watchers — spec `docs/specs/2026-06-16-autonomous-watchers.md`):** the reserved design above is now real. A new pm2 process **`alfred-triggers`** is a *pure scheduler* — it loads enabled `triggers` rows at boot, registers cron schedules on the existing pg-boss queue (and sweeps one-shot `self` triggers for their `next_fire_at`), and enqueues a **`trigger-detect`** job per due tick; it owns timing only and never imports the tool layer (all tool execution stays in the worker, which is where the `Tool` instances + provider already live). The worker's detect handler runs a **tiered detection ladder**: Tier 0 a free deterministic gate (calls a read-tier `Tool`'s `invoke()` directly and diffs the result against `triggers.last_seen_signal`), Tier 1 a cheap-model triage (`DETECTION_MODEL`, `escalate`/`dismiss`), Tier 2 — only on escalation — a full action run created via the unchanged `agent_runs` + `boss.send('agent-run')` path with `human_in_loop=false`. **Detection writes no `agent_runs` row**, so idle polls cost nothing and don't litter the run log. Continuity is the **objective scratchpad** (`memory_facts` scope `trigger:<id>`, folded into the run's `[system]` block, §6.4/§7.7), and the agent's `schedule_self` tool self-schedules future runs (§7.7 seam 3). Because no client is watching when a watcher fires, results/approvals reach the owner via a **durable notifications outbox** (the `notifications` table + the separate `notifications` NOTIFY channel + a worker dispatcher → **Web Push**, the agent-core `Notifier` / `WebPushNotifier`, §7.2/§6.2; deep-linking into the existing `/conversation/:id` chat + approval UI), not SSE — which is dropped on a disconnected client. Deferred (Non-goals): native iOS APNs, other transports (ntfy/Discord DM), the full §7.7 layered objective/daily budget, and a watcher-management UI (v1 manages triggers via `schedule_self` + direct rows).
